Информация от бизнес-заказчика
===

Назначение продукта
---

Продукт позволяет многим пользователям использовать канбан доски по отдельным проектам IT-разработки, смотреть связанные с проектами требования, промежуточные документы аналитики и разработки и отчеты.

Требования
---

1. Продукт должен работать в браузере в виде сайта.
1. Сайт должен поддерживать светлую и темную тему.
1. Сайт должен работать и удобно отображаться на экранах ПК (1024х768, 1920х1080) и смартфонов (Redmi Note 14, iPhone 14+).

1. В продукте должна быть поддержка многих пользователей и их одновременной работы.
1. Изменения осуществляемые разными пользователями в одном проекте должны с минимальной задержкой синхронизироваться между устройствами этих пользователей.
1. Пользователи могут регистрироваться (логин+пароль), восстанавливать доступ по почте, привязывать аутентификаторы для входа по одноразовым паролям TOTP, входить на сайт по логину и паролю или TOTP, выходить из сайта, управлять группами пользователей в своем проекте, добавлять/удалять пользователей в эти группы, управлять правилами доступа этих групп к объектам внутри проекта, просматривать свой тарифный план.
1. У пользователей могут быть разные тарифные планы, влияющие на максимальные количества: собственных проектов (на тарифе по умолчанию - 2), колонок на доске (по умолчанию - 10), исполнителей на доске (по умолчанию - 5). Изменение тарифных планов осуществляется специальным системным пользователем

1. Пользователи могут создавать, удалять, редактировать проекты. У пользователя может быть несколько собственных проектов.
1. Проект содержит список связанных с ним документов, канбан-доску, отчеты о потоке задач на доске.

1. Список связанных с проектом документов:
    - Требования заказчика
    - Требования технического архитектора
    - Требования веб-дизайнера
    - Список user story
    - Описание актуальной архитектуры приложения
    - Список ADR
    - Диаграммы C4: контекста, контейнеров, компонентов
    - Список описаний use case текстом и sequence-диаграммами
    - Описание контрактов, предоставляемых приложением
    - Описание контрактов, используемых приложением
    - Список API тестов с шагами
    - Список UI тестов с шагами
    - Список приемочных тестов с шагами
    - Отчет по актуальным уязвимостям, результам ревью, прохождению тестов и тестовому покрытию

1. Канбан доска содержит задачи, колонки, бэклог, архив, свимлайны, исполнителей.
1. У задач есть признаки - Use Case, Статус (он же - название колонки), Метки, Исполнитель (assignee), Приоритет (low/medium/high/critical). Они отображаются в минималистичной карточке на доске в колонках.
1. У задач есть еще признаки - Название, Описание, Дата создания, Дата взятия в работу, Дата архивирования, Прикрепленные файлы, Комментарии. Они не отображаются в минималистичной карточке, но отображаются при просмотре полной информации о задаче.
1. Задачи могут находиться в одном из трех мест: в бэклоге, в одной из колонок, в архиве. Бэклог находится слева и может быть открыт или скрыт. Архив находится справа и может быть открыт или скрыт.
1. У доски есть фильтры задач по любым из ее признаков. Фильтрация всегда применяется к колонкам; к бэклогу - только если он открыт; к архиву - только если он открыт. При активном фильтре открытие бэклога или архива должно применять к ним активный фильтр.
1. Колонки можно создавать, редактировать, удалять. По умолчанию новая доска содержит три колонки: To Do, In Progress, Done.
1. Свимлайны группируют задачи по Use Case. Задачи с неуказанным Use Case считаются дефектами и попадают в дефолтный свимлайн Bug Fix. Новый свимлайн автоматически появляется при перемещении из бэклога в колонку задачи с Use Case, которому нет соответствия среди существующих свимлайнов. Свимлайн исчезает при перемещении из колонки в архив последней задачи с ее значением Use Case.
1. В бэклоге задачи сортируются по приоритету (более приоритетные - выше)
1. В архиве задачи сортируются по времени перевода в архив (более поздние - выше)
1. В колонках задачи сортируются вручную с помощью drag-n-drop. Порядок должен сохраняться на сервере и быть единым для всех пользователей этой доски.
1. Нажатие на задачу в колонке открывает новую вкладку со страницей с полной информацией по задаче.
1. Между колонками задачи могут перемещаться через drag-n-drop или через редактирования колонки Статус.
1. Исполнители могут быть либо пользователями, либо ботами (подсоединенным к LLM-агенту).
1. Панель исполнителей и связанных с ними задач находится снизу и может быть открыта или скрыта. Нажатие на исполнителя применяет фильтр по выбранному исполнителю. Нажатие на связанную с исполнителем задачу действует аналогично нажатию на задачу в колонке
1. Изменения в признаках задач, перемещение по статусам, добавление, удаление, редактирование комментариев должно логироваться
1. На канбан доске должно быть поле поиска, отправка текста в которое открывает новую вкладку и делает поиск по названию и описанию всех задач проекта с возможностью включить/исключить задачи в бэклоге или архиве, включить/исключить поиск среди комментариев.

1. Отчеты по канбан доске содержат Cumulative Flow Diagram, Lead Time Distribution Chart, диаграмму Гантта по Use Case, лог 100 последних событий этой доски с воможностью подгрузки по 100 предыдущих событий.
