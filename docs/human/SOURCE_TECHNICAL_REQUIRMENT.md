Информация от технического эксперта
===

Стек
---

**Бэк**: Реактивный Kotlin, Spring Boot, Swagger, Postgres, kotlin-logging, Gradle (kotlin dsl), Ktlint, editorconfig, Detekt, Junit, Mockk

**Фронт**: Vue3, Typescript, SCSS, Vite, Vitest, Lcov (покрытие), eslint, prettier

**Сборка и деплой**: Nginx, Docker, Docker Compose, Make

**Подходы**: мультимодульный проект, монорепозиторий, Low Coupling and High Cohesion (LCHC), гексагональная архитектура, DDD, SOLID, активное логирование.

Структура проекта
---
- `docs/ai` - документация по проекту, составленная LLM-агентом.
- `docs/human` - документация по проекту, составленная человеком. LLM-агенту запрещено ее редактировать.
- `spring` - точка входа приложения, настройка фреймворка, авторизации и аутентификации, мониторинга и логирования, объединение воедино всех модулей бэка.
- `usecase` - бизнес-логика
- `webapi` - работа с http, websocket, server sent events
- `postgres` - хранение данных внутри приложения
- `vue` - web ui, работа в браузере
- `nats` - распределенный брокер сообщений
- `playwright` - приемочные тесты на Playwright+TypeScript

Особенности фронтенда
---
1. Модуль фронта должен иметь у себя `build.grale.kts` и в нем сделать соответствие gradle-задач clean, check, assemble, build, test npm-задачам из package.json или описать эту логику самостоятельно. Цель - при очистке, проверках или сборке родительского проекта аналогичные действия происходят и для фронта

1. Клиенты поддерживают HTTP/2

Особенности бэкенда
---
1. Стремимся к Low Coupling (низкая связанность) и High Cohesion (высокое зацепление), стараемся ограничивать видимость классов везде, где это возможно, стремимся располагать классы/интерфейсы данных вложенными в классы/интерфейсы соответствующих операций.

1. Контролируем SOLID.

1. Придерживаемся гексагональной архитектуры и Domain Driven Development, изолируем домены друг от друга и от адаптеров.

1. Используем много профилей Spring. Цель - запускать приложение, независимо указывая профили, меняющие поведение отдельных модулей, с помощью профилей добавлять или удалять feature-toggle.

Валидация кода
---
Для фронта требуется настроить prettier, eslint, eslint-plugin-vue.

Для бэка требуется настроить Ktlint, Detekt.

Запускать валидацию кода и исправлять полученные замечания нужно автоматически после любой порции крупных правок кода.

Тестирование
---
На бэкенде нужно предпочитать юнит-тесты и sliced-тесты (по типу WebFluxTest и DataR2dbcTest) - около 75% от всех тестов бэкенда, остальные тесты - сквозные API (от api до бд) с поднятием всего контекста spring. Jacoco собирает покрытие. 

На фронте нужно предпочитать тесты отдельных компонентов - около 75% от всех тестов фронтенда, остальные тесты - сквозные end-to-end (от web ui до моков api). Lcov собирает покрытие.

Наблюдаемость
---
В коде необходимо использовать логирование, используя разные уровни логирования по примеру:

```kotlin
log.debug('Вызов метода/возврат его результата/одна из развилок в логике. Полезно разработчику для отладки');
log.info('Вызвана операция/ее успешный исход. Полезно бизнес-заказчику');
log.warn('Предусмотренный, но неуспешный исход. Полезно бизнес-заказчику');
log.error('Ошибка, что-то сломалось! Вот информация об ошибке. Полезно разработчику для исправления');
```
При логировании необходимо использовать MDC и класть в него:
- requestId http-запроса (UUID),
- sessionId пользователя (UUID),
- userId пользователя (UUID),
- название http запроса ("<метод> <путь из url>")
- eventId события (UUID),
- название канала, откуда получено событие

Поддерживаемость
---
Классы, data-классы, интерфейсы, методы должны иметь KDoc или JSdoc (документирующие комментарии для kotlin или typescript)
