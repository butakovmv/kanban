# Нефункциональные требования

## 1. Технологический стек

### 1.1 Бэкенд
- Язык: Reactive Kotlin.
- Фреймворк: Spring Boot (реактивный стек).
- Документирование API: Swagger.
- База данных: Postgres.
- Логирование: kotlin-logging.
- Сборщик: Gradle (Kotlin DSL).
- Статический анализ: Ktlint, Detekt.
- Тестирование: JUnit, Mockk.
- Покрытие: Jacoco.

### 1.2 Фронтенд
- Фреймворк: Vue 3.
- Язык: TypeScript.
- Стили: SCSS.
- Сборщик: Vite.
- Тестирование: Vitest.
- Покрытие: Lcov.
- Статический анализ: ESLint, Prettier.

### 1.3 Инфраструктура
- Веб-сервер: Nginx.
- Контейнеризация: Docker, Docker Compose.
- Сборка и оркестрация: Make.

## 2. Архитектура

- **Мультимодульный проект**: код разделён на независимые модули.
- **Монорепозиторий**: все модули в одном репозитории.
- **Low Coupling and High Cohesion (LCHC)**: минимизация зависимостей между модулями, максимизация связности внутри модуля.
- **Гексагональная архитектура**: чёткое разделение на ядро (бизнес-логику) и адаптеры (ввод/вывод).
- **Domain-Driven Design (DDD)**: доменная модель как основа проектирования.
- **SOLID**: принципы объектно-ориентированного проектирования.
- **Активное логирование**: все ключевые операции логируются.

### 2.1 Структура модулей

| Модуль | Назначение |
|---|---|
| `spring` | Точка входа приложения, настройка фреймворка, авторизация, аутентификация, мониторинг, логирование, сборка модулей |
| `usecase` | Бизнес-логика |
| `webapi` | HTTP, WebSocket, Server-Sent Events |
| `postgres` | Хранение данных |
| `vue` | Web UI (браузер) |
| `nats` | Распределённый брокер сообщений |
| `e2etest` | Приёмочные тесты (Playwright + TypeScript) |

### 2.2 Сборка фронта
- Модуль `vue` содержит `build.gradle.kts`.
- Gradle-задачи `clean`, `check`, `assemble`, `build`, `test` мапятся на соответствующие npm-задачи из `package.json`.

## 3. Валидация кода

- **Фронт**: после каждой крупной порции правок запускать Prettier, ESLint (с eslint-plugin-vue).
- **Бэк**: после каждой крупной порции правок запускать Ktlint, Detekt.
- Замечания автоматически исправляются.

## 4. Тестирование

### 4.1 Бэкенд
- ~75% тестов — юнит-тесты и sliced-тесты (WebFluxTest, DataR2dbcTest).
- ~25% тестов — сквозные API-тесты (от API до БД) с поднятием полного контекста Spring.
- Покрытие собирается Jacoco.

### 4.2 Фронтенд
- ~75% тестов — тесты отдельных компонентов.
- ~25% тестов — сквозные end-to-end тесты (от Web UI до моков API).
- Покрытие собирается Lcov.

## 5. Наблюдаемость и логирование

- Использовать уровни логирования:
  - `DEBUG` — вызов метода, возврат результата, развилки в логике (для разработчика).
  - `INFO` — вызвана операция / её успешный исход (для бизнес-заказчика).
  - `WARN` — предусмотренный, но неуспешный исход (для бизнес-заказчика).
  - `ERROR` — ошибка / исключительная ситуация (для разработчика).
- В MDC логирования обязательно помещать:
  - `requestId` — UUID HTTP-запроса;
  - `sessionId` — UUID сессии пользователя;
  - `userId` — UUID пользователя;
  - название HTTP-запроса (`<МЕТОД> <путь>`);
  - `eventId` — UUID события;
  - название канала, откуда получено событие.

## 6. Совместимость

- **Браузер**: продукт работает в браузере (веб-сайт).
- **Темы**: поддержка светлой и тёмной темы.
- **Экраны ПК**: 1024×768, 1920×1080.
- **Экраны смартфонов**: Redmi Note 14, iPhone 14+ (и аналогичные).
- Адаптивная вёрстка для комфортного отображения на всех указанных разрешениях.

## 7. Производительность

- Одновременная работа многих пользователей.
- Минимальная задержка синхронизации изменений между пользователями одного проекта.

## 8. Безопасность

- Хранение паролей с использованием современных алгоритмов хеширования.
- Поддержка TOTP как альтернативы паролю для входа в систему.
- Разграничение доступа к объектам проекта через группы пользователей.
- Восстановление доступа через подтверждение по электронной почте.

## 9. Качество кода

- Использование `editorconfig` для единого стиля кода.
- Применение Ktlint и Detekt для бэкенда.
- Применение ESLint и Prettier для фронтенда.
- Автоматический запуск валидации после крупных правок.
