# Функциональные требования

## 1. Аутентификация и управление пользователями

### 1.1 Регистрация и вход
- Пользователь может зарегистрироваться по логину и паролю.
- Пользователь может войти на сайт по логину и паролю.
- Пользователь может войти на сайт по логину и одноразовому коду TOTP (вместо пароля).
- Пользователь может восстановить доступ к учётной записи через почту.
- Пользователь может привязать TOTP-аутентификатор для входа по одноразовым паролям.
- Пользователь может отвязать TOTP-аутентификатор.
- Пользователь может выйти из сайта.

### 1.2 Управление группами и доступом
- Владелец проекта может управлять группами пользователей внутри своего проекта.
- Владелец проекта может добавлять и удалять пользователей в/из групп.
- Владелец проекта может управлять правилами доступа групп к объектам внутри проекта.

### 1.3 Тарифные планы
- Пользователь может просматривать свой тарифный план.
- Тарифный план влияет на максимальное количество:
  - собственных проектов (по умолчанию — 2);
  - колонок на доске (по умолчанию — 10);
  - исполнителей на доске (по умолчанию — 5).
- Изменение тарифного плана может выполнять только специальный системный пользователь.

## 2. Управление проектами

- Пользователь может создавать, редактировать и удалять проекты.
- У пользователя может быть несколько собственных проектов (в пределах тарифного плана).
- Проект содержит:
  - список связанных документов;
  - канбан-доску;
  - отчёты о потоке задач на доске.

## 3. Управление документами проекта

Проект может содержать следующие типы связанных документов:
- Требования заказчика;
- Требования технического архитектора;
- Требования веб-дизайнера;
- Список user story;
- Описание актуальной архитектуры приложения;
- Список ADR (Architecture Decision Records);
- Диаграммы C4: контекста, контейнеров, компонентов;
- Список описаний use case текстом и sequence-диаграммами;
- Описание контрактов, предоставляемых приложением;
- Описание контрактов, используемых приложением;
- Список API-тестов с шагами;
- Список UI-тестов с шагами;
- Список приёмочных тестов с шагами;
- Отчёт по актуальным уязвимостям, результатам ревью, прохождению тестов и тестовому покрытию.

## 4. Канбан-доска

### 4.1 Структура доски
- Доска содержит: колонки, бэклог, архив, свимлайны, исполнителей.
- По умолчанию новая доска содержит три колонки: To Do, In Progress, Done.
- Колонки можно создавать, редактировать и удалять.
- Бэклог находится слева от колонок и может быть открыт или скрыт.
- Архив находится справа от колонок и может быть открыт или скрыт.
- Панель исполнителей находится снизу доски и может быть открыта или скрыта.

### 4.2 Свимлайны (frontend-only)
- Свимлайны группируют задачи по полю Use Case. Управление свимлайнами — исключительно клиентская логика; сервер не хранит и не обрабатывает свимлайны.
- Задачи с неуказанным Use Case считаются дефектами и попадают в дефолтный свимлайн «Bug Fix».
- Фронт автоматически создаёт свимлайн при появлении задачи с новым Use Case в колонках.
- Фронт автоматически удаляет свимлайн, когда последняя задача с этим Use Case покидает колонки.

### 4.3 Задачи
- Задача имеет следующие признаки, отображаемые в минималистичной карточке на доске:
  - Use Case;
  - Статус (название колонки);
  - Метки;
  - Исполнитель (assignee);
  - Приоритет (low / medium / high / critical).
- Задача имеет дополнительные признаки, отображаемые при просмотре полной информации:
  - Название;
  - Описание;
  - Дата создания;
  - Дата взятия в работу;
  - Дата архивирования;
  - Прикреплённые файлы;
  - Комментарии.
- Задача может находиться в одном из трёх мест: бэклог, одна из колонок, архив.
- Нажатие на задачу в колонке открывает новую вкладку со страницей полной информации по задаче.

### 4.4 Сортировка задач
- В бэклоге задачи сортируются по приоритету (более приоритетные выше).
- В архиве задачи сортируются по времени перевода в архив (более поздние выше).
- В колонках задачи сортируются вручную с помощью drag-n-drop. Порядок сохраняется на сервере и един для всех пользователей доски.

### 4.5 Перемещение задач
- Задачи перемещаются между колонками через drag-n-drop.
- Задачи перемещаются между колонками через изменение поля «Статус».

### 4.6 Исполнители
- Исполнителем может быть пользователь или бот (подключённый LLM-агент).
- Нажатие на исполнителя в панели исполнителей применяет фильтр по выбранному исполнителю.
- Нажатие на задачу, связанную с исполнителем, действует аналогично нажатию на задачу в колонке.

## 5. Фильтрация и поиск

### 5.1 Фильтры доски
- Доска поддерживает фильтры задач по любым из её признаков.
- Фильтрация всегда применяется к колонкам.
- Фильтрация применяется к бэклогу, только если он открыт.
- Фильтрация применяется к архиву, только если он открыт.
- При активном фильтре открытие бэклога или архива должно применять к ним активный фильтр.

### 5.2 Поиск
- На канбан-доске есть поле поиска.
- Отправка текста в поле поиска открывает новую вкладку с результатами поиска.
- Поиск выполняется по названию и описанию всех задач проекта.
- Пользователь может включить/исключить задачи из бэклога.
- Пользователь может включить/исключить задачи из архива.
- Пользователь может включить/исключить поиск среди комментариев.

## 6. Отчёты

Доска предоставляет следующие отчёты:
- Cumulative Flow Diagram (CFD);
- Lead Time Distribution Chart;
- Диаграмма Гантта по Use Case;
- Лог последних 100 событий доски с возможностью подгрузки по 100 предыдущих событий.

## 7. Логирование изменений

- Изменения признаков задач, перемещение по статусам, добавление/удаление/редактирование комментариев логируются.

## 8. Совместная работа (real-time)

- Изменения, осуществляемые разными пользователями в одном проекте, должны с минимальной задержкой синхронизироваться между устройствами этих пользователей.
