# Протоколы и технологии взаимодействия фронта и бэка

## Контекст
Приложение — мультимодульная канбан-доска с реальным временем. Фронт (Vue 3) и бэк (Reactive Kotlin / Spring Boot) обмениваются данными для аутентификации, CRUD-операций, синхронизации досок, загрузки файлов и отчётов. Необходимо выбрать протоколы и технологии для каждого типа операций.

## Анализ операций

### 1. CRUD-операции (JSON over HTTP)
| Группа | Эндпоинты | Частота | Размер |
|---|---|---|---|
| Аутентификация | login, register, recover, TOTP, logout, session | Низкая | < 1 KB |
| Проекты | list, create, update, delete | Средняя | < 5 KB |
| Колонки | create, update, delete | Средняя | < 1 KB |
| Задачи | create, update, delete, move, reorder, archive | Высокая | < 5 KB |
| Комментарии | list, create, update, delete | Средняя | < 2 KB |
| Документы | list, delete | Низкая | < 10 KB |
| Группы/права | create, update, delete, members | Низкая | < 2 KB |
| Администрирование | user search, tariff change | Очень низкая | < 2 KB |
| Поиск | search | Средняя | < 50 KB |
| Отчёты | chart data, event log | Низкая | 10–500 KB |

**Характер:** синхронные запрос-ответ, идемпотентность разная (GET — да, POST — нет, PUT — да, DELETE — да).

### 2. Операции реального времени
| Событие | Направление | Частота | Задержка |
|---|---|---|---|
| Перемещение задачи | A → бэк → B | Высокая | < 100 мс |
| Изменение полей задачи | A → бэк → B | Средняя | < 100 мс |
| Создание/удаление колонки | A → бэк → B | Низкая | < 200 мс |
| Создание задачи | A → бэк → B | Средняя | < 200 мс |
| Изменение порядка | A → бэк → B | Высокая | < 100 мс |

**Характер:** асинхронная публикация событий от одного клиента всем остальным, подключённым к тому же проекту. Требуется постоянное соединение.

### 3. Файловые операции
| Операция | Направление | Размер | Частота |
|---|---|---|---|
| Загрузка файла (задача/документ) | Клиент → сервер | до 10 MB | Низкая |
| Скачивание файла | Сервер → клиент | до 10 MB | Низкая |

**Характер:** большие бинарные данные, потоковая передача, важна индикация прогресса.

## Решение

### 1. REST API (Reactive HTTP + JSON)
Для всех CRUD-операций используется **REST over HTTP** (браузеры поддерживают HTTP/2; сервер отдаёт предпочтение HTTP/2 через ALPN).

- **Фреймворк:** Spring WebFlux (`@RestController`, `RouterFunction`)
- **Формат:** JSON (application/json), для ошибок — единая схема `{ "error": "...", "detail": "...", "code": "..." }`
- **Методы:** GET (чтение), POST (создание), PUT (полное обновление), PATCH (частичное), DELETE (удаление)
- **Статусы:** 200 OK, 201 Created, 204 No Content, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 500 Internal Server Error
- **Пагинация:** `?offset=0&limit=50` + заголовок `X-Total-Count` или тело `{ "items": [...], "total": N }`
- **Аутентификация:** Bearer JWT в заголовке `Authorization`
- **MDC:** requestId, sessionId, userId, httpMethod+path — передаются через заголовки или извлекаются из JWT

#### Почему REST, а не GraphQL
- Операции хорошо ложатся на ресурсную модель
- Проще кеширование на уровне HTTP
- Меньше сложности для фронта (не нужен клиент GraphQL)
- Достаточно одного запроса на экран

### 2. SSE для получения обновлений доски (REST + SSE)
Для синхронизации досок в реальном времени используется **Server-Sent Events** (SSE). Отправка действий — через стандартные REST-запросы.

**Архитектура:**
- Клиент выполняет все мутирующие операции через REST (PUT/POST/DELETE) — как и любые другие CRUD-операции
- Сервер после обработки REST-запроса публикует событие во **внутренний event bus**
- Все SSE-соединения, подписанные на проект, получают событие
- Клиент получает обновления по единственному SSE-каналу без дополнительных запросов

```
┌──────────┐   REST (PUT /tasks/:id/move)    ┌──────────┐
│ Клиент A │ ──────────────────────────────→ │  Сервер  │
│          │                                 │          │
│          │ ←── SSE (event: task_moved) ─── │  (event  │
│          │                                 │   bus)   │
│ Клиент B │ ←── SSE (event: task_moved) ─── │          │
└──────────┘                                 └──────────┘
```

**Реализация на сервере:**
- **SSE-эндпоинт:** `GET /api/v1/projects/{id}/events` — возвращает `Flux<ServerSentEvent<ProjectEvent>>` (Spring WebFlux)
- **Event bus:** `Sinks.Many<ProjectEvent>` (Project Reactor) — in-memory шина событий на один экземпляр
- **Подписка:** каждый SSE-клиент создаёт подписку на `Sinks`, фильтруя по projectId; при отключении клиента подписка отменяется
- **Формат события:** `{ "type": "TASK_MOVED" | "TASK_UPDATED" | "COLUMN_CREATED" | ..., "payload": {...}, "userId": "...", "timestamp": "..." }`
- **Last-Event-ID:** SSE-native — браузер передаёт `Last-Event-ID` при переподключении; сервер хранит буфер последних N событий для восполнения пропущенных

**Реализация на клиенте:**
- **EventSource** — встроенный браузерный API, не требует дополнительных библиотек
- Авто-переподключение: браузер сам восстанавливает SSE-соединение с `Last-Event-ID`
- Полученные события применяются к локальному состоянию доски (Pinia store)

**Кластеризация:** In-memory `Sinks` при необходимости заменяется на NATS или другой брокер — это внутренняя деталь инфраструктуры бэкенда и не влияет на контракт с фронтом.

**Отказоустойчивость:**
- При разрыве SSE: `EventSource` автоматически переподключается с exponential backoff (встроено в браузер)
- При длительном разрыве (> 30 сек): клиент выполняет полную перезагрузку доски через REST GET
- Буфер последних событий на сервере позволяет не пропустить события между отключением и переподключением

### 3. Multipart HTTP для файлов
Для загрузки файлов используется **стандартный multipart/form-data** поверх HTTP.

- **Загрузка:** POST с `Content-Type: multipart/form-data`, обработка через Spring WebFlux `Part` / `FilePart`
- **Прогресс:** на клиенте через `XMLHttpRequest.upload.onprogress` или `axios.onUploadProgress`
- **Размер:** ограничение 10 MB (настраивается в Nginx + Spring)
- **Скачивание:** GET с заголовком `Content-Disposition: attachment; filename="..."`, потоковый response (Spring `DataBuffer` / `Resource`)
- **Хранение:** файлы сохраняются на локальной файловой системе или S3-совместимом хранилище (MinIO); в БД — только метаданные и путь

## Альтернативы, которые были рассмотрены

### gRPC
- Отклонён: избыточен для браузерного фронта (нет нативной поддержки в браузерах, нужен gRPC-Web + прокси)
- Плюсы: строгая типизация, бинарный протокол, streaming

### Polling вместо WebSocket
- Отклонён: даёт задержку от 1 секунды (частый poll) до высокой нагрузки на сервер (редкий poll)
- Неприемлемо для сценария drag-n-drop, где изменение нужно показать другим пользователям сразу

### WebSocket (как альтернатива SSE)

**Как работает:** клиент и сервер устанавливают постоянное двустороннее соединение через протокол ws://. Для маршрутизации сообщений используется STOMP поверх WebSocket.

**Плюсы:**
- **Дуплекс** — клиент отправляет действия и получает события по одному соединению
- **Меньшая задержка** — действие уходит сразу по открытому сокету без HTTP-накладных расходов
- **Неограниченное число соединений** — WebSocket использует отдельный протокол, не привязан к лимитам HTTP/1.1
- **Бинарные данные** — можно передавать бинарные фреймы (не критично для JSON-событий)

**Минусы:**
- **Ручное переподключение** — нет встроенного авто-reconnect, требуется SockJS или собственная реализация
- **STOMP-зависимость** — нужна библиотека STOMP.js на клиенте и STOMP-конфигурация на сервере
- **Блокировка прокси** — корпоративные прокси могут блокировать ws:// upgrade handshake
- **Сложнее отладка** — WebSocket не виден в инструментах разработчика как HTTP (только отдельная вкладка)

**Почему отклонён в пользу SSE:**
Канбан-доска — не чат и не игра. Частота действий — несколько в минуту (drag-n-drop), а не сотни в секунду. Накладные расходы HTTP-запроса (заголовки, JWT) — единицы килобайт — незначимы при такой частоте. При этом SSE даёт:
- встроенное авто-переподключение (браузерный `EventSource`)
- нулевую зависимость от дополнительных библиотек (ни STOMP, ни SockJS не нужны)
- единый event bus на сервере без STOMP-конфигурации
- прохождение через любые прокси

Разница в latency между WS и REST+SSE (10–50 мс против 20–100 мс) для канбан-доски незаметна пользователю.

## Последствия
- **REST + SSE проще, чем REST + WebSocket:**
  - Сервер: `Flux<ServerSentEvent>` + `Sinks.Many` — без STOMP, без WebSocketHandler, без heartbeat
  - Клиент: `EventSource` (встроен в браузер) — без STOMP.js, без SockJS, без reconnect-логики
- **Единый канал событий** — все обновления доски приходят по одному SSE-соединению; клиент не подписывается на разные топики
- **Авто-переподключение** `EventSource` с `Last-Event-ID` встроено в браузер — не нужно писать и тестировать reconnect-логику
- **REST остаётся единственным способом мутации данных** — все операции (CRUD + перемещение задач) идут через HTTP; это единообразно и предсказуемо
- **Единый формат JWT** для REST и SSE упрощает аутентификацию
- **Файлы** передаются через отдельные multipart-запросы — это разделяет каналы и упрощает отладку
- **HTTP/2** снимает лимит соединений для SSE, но если сервер работает на HTTP/1.1 за Nginx — Nginx проксирует SSE без проблем (буферизация отключается через `X-Accel-Buffering: no`)
