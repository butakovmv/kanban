Aдаптер приложения к web API
===

Структура
---
В корневом пакете находятся пакеты, соответствующие доменным областям приложения. В них находятся классы-мапперы, преобразующие специфичные для этого модуля dto в связанные с доменами и обратно, а также классы-обработчики, объединяющие в себе маппинг, обращение к `usecase`, анализ пользователя и прав доступа.

В корневом пакете находятся пакеты `http`, `websocket`, `sse`. В них находятся подготовка и настройка этих технологий, классы конфигурации, исключений,  контроллеров со специфичными для технологии аннотациями и настройками. Контроллеры в этих пакетах ответственны только за прием сообщения из сети, и общение с соответствующим обработчиком из доменных пакетов этого модуля.

Требования к структуре кода
---
Все классы модуля должны иметь видимость `internal` или `private`.

Разрешается объединять вместе интерфейс/класс и используемые в нем интерфейсы/классы данных как вложенные интерфейсы/классы, если они используется только вместе.

Контроллеры обязаны описывать только один запрос на класс/файл.

Крайне нежелательно обращаться к классам одной предметной области из другой предметной области.

Тесты
---
В модуле необходимо использовать только:
- юнит-тесты для мапперов, конфигурации и служебных классов
- `@WebFluxTest` в связке с `WebTestClient` для тестирование тел запросов и ответов, корректности кодов ответа, корректности прав доступа для выбранного пользователя

Тесты с поднятием полного Spring-контекста в модуле запрещены.

Необходимо использовать генераторы, которые создают случайные тестовые dto, с которыми будет работать тест, и передают их в моки интерфейсов из модуля `usecase`.
Например, `SomeRequestGenerator` — генерирует `SomeRequest`, служащий телом для запроса `someRequest`.
